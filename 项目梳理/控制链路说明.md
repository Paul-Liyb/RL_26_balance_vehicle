# 强化学习平衡车控制链路说明

## 1. 当前结论

- 这套修改版固件的主控制入口是 `MiniBalance/control.c` 中的 `HAL_GPIO_EXTI_Callback()`。
- MPU6050 中断每 `5 ms` 触发一次采样；控制量真正更新一次是每 `10 ms`。
- 当前 LQR 的状态维度是 `8`，动作维度是 `2`：
  - 状态顺序：`theta_L, theta_R, theta_1, theta_2, theta_L_dot, theta_R_dot, theta_dot_1, theta_dot_2`
  - 动作顺序：`u_L, u_R`
- `u_L/u_R` 的物理意义不是 PWM，而是左右轮角加速度 `rad/s^2`；固件会先把它积分成目标轮速，再交给 PI 速度环，最后变成 PWM。
- `Control_mode == 0` 时使用板载 LQR。
- `Control_mode != 0` 时使用外部输入 `val1/val2`，当前 `control.h` 里宏定义的是 `#define Control_mode 1`，也就是默认走外部动作分支。
- MATLAB 脚本离散化并 `dlqr` 求出的 `K`，与 `control.c` 和 `wifi3.0.py` 里的现有系数一致，最大绝对误差约为 `4.35e-05`。

## 2. 整体数据流

```text
MPU6050 INT (5 ms)
  -> HAL_GPIO_EXTI_Callback()
  -> 读取编码器 / 摆杆 ADC / 姿态
  -> 组装状态量
  -> 每 10 ms 计算一次控制输入 u_L/u_R
     -> Control_mode == 0: 板载 LQR
     -> Control_mode != 0: 使用 USART3 收到的 val1/val2
  -> TargetVal_L/TargetVal_R = theta_dot + u * 0.01
  -> Incremental_L/R() 做速度 PI
  -> PWM_Limit()
  -> Set_Pwm()
  -> 电机驱动
```

如果使用上位机闭环，则还有一条通信回路：

```text
固件 APP_Show()
  -> USART3 printf 输出 13 个浮点状态
  -> WiFi 模块 TCP 转发到 PC
  -> wifi3.0.py 解析状态并计算 u_L/u_R
  -> 发送二进制动作包: 0xAA + float(u_L) + float(u_R) + checksum
  -> USART3 中断解析后写入 val1/val2
  -> control.c 在下一次 10 ms 控制时取用
```

## 4. 固件内部控制链路

### 4.1 采样与状态更新

`HAL_GPIO_EXTI_Callback()` 由 MPU6050 的中断脚触发，每 `5 ms` 执行一次：

1. 读取左右编码器。
2. 读取摆杆 ADC `Angle_ADC`。
3. 调用 `Get_Angle()` 更新 `Angle_Balance` 和陀螺仪角速度。
4. 计算轮瞬时角速度：
   - `theta_L_dot_instant`
   - `theta_R_dot_instant`
5. 积分得到轮角度：
   - `theta_L`
   - `theta_R`
6. 计算车身倾角与角速度：
   - `theta_1 = Angle_Balance / 180 * pi`
   - `theta_dot_1` 由 `Gyro_Balance` 累加后在 10 ms 节拍取平均
7. 每两个 5 ms 周期，也就是每 `10 ms`，补全慢变量：
   - `theta_L_dot = (theta_L - last_theta_L) / 0.01`
   - `theta_R_dot = (theta_R - last_theta_R) / 0.01`
   - `theta_2 = (Angle_Balance + angle_count(Angle_ADC, mid)) / 180 * pi`
   - `theta_dot_2 = (theta_2 - last_theta_2) / 0.01`，再做一阶低通

### 4.2 当前 LQR 使用的状态

固件里 LQR 的状态顺序就是：

1. `theta_L`
2. `theta_R`
3. `theta_1`
4. `theta_2`
5. `theta_L_dot`
6. `theta_R_dot`
7. `theta_dot_1`
8. `theta_dot_2`

对应控制律：

```text
u = -K (x - x_target)
```

其中 `K` 是 `2x8`，动作是：

1. `u_L`
2. `u_R`

### 4.3 参考量和一个容易忽略的细节

`Normal()` 会根据遥控/避障逻辑修改：

- `Target_theta_L`
- `Target_theta_R`
- `Target_theta_L_dot`
- `Target_theta_R_dot`
- `Target_theta_1`

但函数结尾又会把下面三个量统一清零：

- `Target_theta_L_dot = 0`
- `Target_theta_R_dot = 0`
- `Target_theta_1 = 0`

所以当前这套代码里，真正持续生效的参考量主要是：

- `Target_theta_L`
- `Target_theta_R`

这也解释了为什么 `wifi3.0.py` 直接把 5 个 target 量都置零后仍能复现现有 LQR。

### 4.4 动作到 PWM 的链路

控制器输出 `u_L/u_R` 后，固件继续做：

1. `TargetVal_L = theta_L_dot + u_L * 0.01`
2. `TargetVal_R = theta_R_dot + u_R * 0.01`
3. `Incremental_L/R()` 速度 PI
4. `PWM_Limit()` 限幅到 `[-6900, 6900]`
5. `Set_Pwm()` 写入左右电机 PWM

因此以后做 RL 时，若仍沿用现有接口，RL 输出应继续对齐为角加速度 `u_L/u_R`，而不是直接输出 PWM。

## 5. 通信关系：wifi3.0.py 与固件

### 5.1 固件到 PC

- `main.c` 中主循环在 `answer_flag == true` 时调用 `APP_Show()`。
- `APP_Show()` 通过 `printf` 发送一帧文本：

```text
{theta_1:theta_dot_1:theta_2:theta_dot_2:theta_L:theta_R:theta_L_dot:theta_R_dot:Target_theta_L:Target_theta_R:Target_theta_L_dot:Target_theta_R_dot:Target_theta_1}
```

- `usart.c` 里 `fputc()` 已重定向到 USART3。
- `Flag_Show == 0` 时，`printf` 走 USART3；当前默认就是 `0`。
- PDF 说明中给出的链路是：车上 USART3 -> WiFi 模块 -> TCP Server `192.168.4.1:6390`。

### 5.2 PC 到固件

`wifi3.0.py`：

1. 连接 `192.168.4.1:6390`
2. 接收上面的文本状态帧
3. 解析 13 个浮点
4. 取前 8 个状态做 LQR
5. 发回二进制动作包：

```text
0xAA + float(u_L) + float(u_R) + checksum
```

固件侧 `usart3.c` 的 USART3 中断分支会：

1. 查找包头 `0xAA`
2. 累积 10 字节
3. 校验和匹配后解析两个 `float`
4. 写入：
   - `val1`
   - `val2`
5. 置 `answer_flag = true`

### 5.3 当前握手机制

当前闭环不是“严格固定频率的主从同步”，而是一个基于 `answer_flag` 的握手机制：

- 上电初始 `answer_flag = true`
- 主循环发送一次状态后，把它清成 `false`
- PC 返回动作后，USART3 中断把它置回 `true`
- 主循环再发送下一帧状态
- `control.c` 在 `Control_mode != 0` 分支里读取 `val1/val2`

这能工作，但它也意味着：

- 上位机闭环节拍取决于通信往返，而不是单纯固定 `100 Hz`
- `answer_flag` 同时被主循环和中断/控制逻辑使用，后面做 RL 训练时最好把“状态发送”和“动作生效”的时序关系再理顺得更明确

## 6. 当前 LQR 频率、状态、动作

| 项目 | 当前实现 |
| --- | --- |

| 动作维度 | `2` |
| 动作含义 | 左右轮角加速度 `u_L/u_R` |

## 7. RL 应该替换/接入的位置

### 方案 A：继续用 PC 端控制

最小改动方案是保留当前固件，只替换 `wifi3.0.py` 的控制器：

- 保留固件的状态发送
- 保留 USART3 收动作
- 把 `wifi3.0.py` 中 LQR 计算替换成 RL actor 推理

这种方式的优点：

- 改固件最少
- 便于先在 PC 上迭代模型
- 容易做日志记录

限制：

- 受 WiFi 往返和 `answer_flag` 握手影响
- 不适合追求严格实时性

### 方案 B：把 RL 推理放到板上

固件里已经预留了：

- `RL_Model_Init()`
- `RL_Controller()`

最自然的板载接入点就是 `control.c` 的 `count == 2` 那个 `10 ms` 控制块。也就是：

1. 先组装好 8 维状态
2. 在这里调用 RL 推理
3. 直接写 `u_L/u_R`
4. 复用后面的积分、PI、PWM 部分

如果做板载 RL，建议不要改后级速度环，先保持“RL 输出角加速度”这个接口不变。

## 8. 后面做 RL 训练时最可能要改的地方

### 必改文件

- `MiniBalance/control.c`
  - 决定状态怎么组装
  - 决定动作是在板上算还是在外部送进来
- `MiniBalance/show.c`
  - 决定上位机能看到哪些状态
- `HAREWARE/usart3.c`
  - 决定动作包格式、校验与同步
- `MiniBalance/Inc/control.h`
  - `Control_mode` 当前是编译期宏，后面可能需要改成运行时切换
- `平衡车补充资料/wifi3.0.py`
  - 如果走 PC 端 RL，这是最直接的替换点

### 高概率还要补的内容

- 更明确的 step 同步机制
  - 最好把“发状态”“收动作”“应用动作”拆成更清晰的状态机
- 超时保护
  - 外部动作长时间没回来时，应该回退到安全动作或停机
- 训练日志
  - 时间戳、状态、动作、角度越界、急停原因
- reset/episode 逻辑
  - 目前代码更像在线控制，不像训练环境
- 动作裁剪
  - 现在后级只对 PWM 限幅，没有单独对 RL 动作做策略层裁剪

## 9. 与补充 PDF 的对应关系

PDF 里的关键信息与代码已能对应上：

- 目标就是“平衡车经 WiFi 把状态发给 PC，PC 算动作再回传”
- 代码路径明确指向这个修改版工程
- `wifi3.0.py` 被说明为 LQR 等效公式示例
- 发送状态通过 `APP_Show()`
- 接收动作通过 `usart3.c` 的 `HAL_UART_RxCpltCallback()`
- 外部动作最终在 `control.c` 的控制分支中生效

## 10. 当前可直接复用给 RL 的 I/O 定义

如果你后面先做一个最小可用 RL 闭环，建议先完全沿用这组定义：

- 观测 `obs`：
  - `theta_L`
  - `theta_R`
  - `theta_1`
  - `theta_2`
  - `theta_L_dot`
  - `theta_R_dot`
  - `theta_dot_1`
  - `theta_dot_2`
- 动作 `act`：
  - `u_L`
  - `u_R`

这样可以最大程度复用现有 LQR 链路，并与 `tools/lqr_from_matlab.py` 的结果直接对比。
