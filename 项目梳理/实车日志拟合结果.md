# 实车日志拟合结果

数据源：`小组资料/balance_data.csv`

该文件是实车 LQR 闭环采集日志，当前接口按实际 CSV 为准：

```text
state = [
  theta_L,
  theta_R,
  theta_1,
  theta_2,
  theta_L_dot,
  theta_R_dot,
  theta_dot_1,
  theta_dot_2
]

action = [
  u_L,
  u_R
]
```

## 复现命令

```bash
./start_rl_env.sh python3 fit_real_log_model.py
```

默认输出目录：

```text
tools/artifacts/model_fit/
```

主要输出：

- `model_fit_summary.json`
- `model_fit_metrics.csv`
- `real_log_fit_parameters.npz`

这些文件属于本地实验产物，不提交 Git。

## 数据规模

- CSV 行数：`7107`
- 相邻行 transition 数：`7106`
- 训练 transition：`5684`
- 测试 transition：`1422`
- 日志频率：约 `9.5 Hz`
- 默认一步预测间隔：`0.105263 s`

按左右轮角度和速度估计的记录间隔：

```text
median: 0.09467 s
mean:   0.11142 s
p10:    0.03733 s
p90:    0.20363 s
```

说明日志间隔存在抖动，不能当成严格等间隔高频控制数据。

| --- | ---: |
| persistence | 0.446923 |

## 结论

1. 直接用物理模型按 `0.105263 s` 做一步预测，对不上实车日志。
2. 这不等价于物理参数完全错误，因为 CSV 是低频 LQR 闭环日志，中间高频控制过程没有记录。
3. 数据驱动的闭环模型能明显降低一步预测误差。
4. `x -> x_next` 和 `x,u -> x_next` 效果几乎一样，说明 `u_L/u_R` 基本由当前状态决定，符合 LQR 闭环日志特征。
5. `measured_estimate + delta fit` 可以作为后续“实车日志校正模型”的起点，但不能解释成真实质量、重心或转动惯量辨识结果。

## 后续建议

短期：

- 用 `fit_real_log_model.py` 的输出作为行为克隆和实车日志模型分析依据。
- 保留 `measured_estimate` 作为物理先验。
- 已接入两个环境 profile：`real_log_fit` 和 `measured_delta_fit`。
- 使用这些 profile 前先运行 `./start_rl_env.sh python3 fit_real_log_model.py` 生成 `tools/artifacts/model_fit/real_log_fit_parameters.npz`。

示例：

```bash
./start_rl_env.sh python3 train.py \
  --algo sac \
  --model-profile real_log_fit \
  --device cpu \
  --output-dir artifacts/real_log_fit_smoke \
  --seeds 0 \
  --timesteps 10000 \
  --eval-freq 5000 \
  --eval-episodes 5
```

`real_log_fit` 直接使用 `x,u -> x_next` 的日志拟合模型。

`measured_delta_fit` 使用 `measured_estimate` 物理模型加日志残差校正。

如果要做物理参数辨识：

- 需要更高频日志，最好接近控制周期。
- 需要记录每个控制周期的状态和动作。
- 需要开环或半开环激励数据，否则闭环 LQR 会让 `u` 与 `x` 高度相关，参数不可辨识性较强。
